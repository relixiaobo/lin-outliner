# Internationalization (i18n)

How the app's interface text is localized and how the display language is chosen,
applied, and persisted. The migration of strings is ongoing (foundation + slice
shipped); see [`../plans/i18n-multi-language.md`](../plans/i18n-multi-language.md)
for rollout status and the PM-ratified direction.

## Model

A **lightweight, self-built typed** layer — no `i18next`/`react-intl`, no runtime
dependency. English is the canonical message tree and defines the `Messages` type;
every other locale is a deep-partial that falls back to English per key. The typed
dictionary makes a missing or mistyped key a compile error.

- **`src/core/locale.ts`** — `Locale` (a union; every value is file-backed),
  `DEFAULT_LOCALE` (`'en'`), `SUPPORTED_LOCALES` (each labelled with its autonym for
  the picker), `isLocale`, `resolveSystemLocale` (OS tag → nearest supported), and
  `LIN_LANGUAGE_CHANGED_CHANNEL`.
- **`src/core/i18n/messages/en.ts`** — the canonical tree; `type Messages =
  typeof en`. Static strings are plain values; strings with runtime values are arrow
  functions taking a typed params object (`help: ({ app }) => \`${app} Help\``), so
  interpolation is type-checked at the call site.
- **`src/core/i18n/messages/<locale>.ts`** — a `DeepPartial<Messages>`; any omitted
  key renders in English.
- **`src/core/i18n/index.ts`** — `getMessages(locale)` deep-merges the locale's
  override onto English (strings/functions are leaves, only objects recurse) and
  memoizes per locale. Shared by both main and renderer processes.
- **Plurals / number / date** — wrap native `Intl.PluralRules` /
  `Intl.NumberFormat` / `Intl.DateTimeFormat` (built-in CLDR data covers every
  language). Added on first use, not kept as unused scaffolding.

## Selection, application, persistence

The display language is a UI preference stored in `app-preferences.json` alongside
the theme (`src/main/appPreferences.ts`, `language` field; `null` = no explicit pick
yet). The **effective locale** is the stored pick, else `resolveSystemLocale` of the
OS locale.

Theme can rely on Electron's `nativeTheme.themeSource` → `prefers-color-scheme` to
update every renderer for free; **language has no equivalent**, so the main process
drives it explicitly. On `lin:set-language` (`src/main/main.ts`): validate → persist
→ send `LIN_LANGUAGE_CHANGED_CHANNEL` to every window → rebuild the native menu
(`buildApplicationMenu` reads the effective locale). The native application menu,
the spellcheck context menu's "Add to Dictionary", the settings + provider-config
window titles, and the native "insert image" file picker (title + filter label) are
all localized via `getMessages(effectiveLocale())`.

**The whole menu bar follows the app language** (PM decision 2026-06-04: the in-app
language picker wins over the macOS-native convention of OS-language role menus). A
`role` left bare takes its *title* from the OS language, so `buildApplicationMenu`
expands the `editMenu` / `windowMenu` roles into their documented item lists and
gives every role item — plus the View items and the Help-menu title — an explicit
`label` from the effective locale. The roles still carry the native behavior,
accelerators, and enable state. Two things stay OS-localized by design: the dynamic
`togglefullscreen` title (a static label would freeze its "Enter" ⇄ "Exit Full
Screen" toggle) and the items macOS injects itself (the Services submenu, Emoji &
Symbols / Start Dictation, and the live Window list). In a dev run the bold app-menu
title reads "Electron" (the dev bundle name); a packaged build supplies "Tenon".

### No-flash startup

Preload reads the effective locale **synchronously** at load
(`ipcRenderer.sendSync('lin:get-language-sync')`, exposed as
`window.lin.initialLanguage`). The renderer's `I18nProvider`
(`src/renderer/i18n/I18nProvider.tsx`) seeds its first render from it, so launch
never flashes English before the chosen language — the same before-first-paint
discipline the theme uses.

### Renderer usage

Each renderer entry (main app, launcher, settings) wraps its tree in
`<I18nProvider>`. Components read strings with `useT()` (`t.menu.settings`,
`t.menu.help({ app })`); the settings picker uses `useI18n().setLocale`, which
optimistically updates and calls `window.lin.setLanguage`. The provider subscribes
to `onLanguageChanged` so a change made in the settings window updates the main
window and launcher without a reload, and keeps `document.documentElement.lang`
current for a11y and font shaping. The context defaults to English, so `useT()` never
throws when read outside a provider (isolated tests degrade to English).

**Pure helpers** that run outside React (e.g. `launcherModel`, `AgentProcessBlock`'s
`summarizeProcess`, `definitionConfig`, `NodePanel`'s day-title formatter, the
search-query summary/outline builders) cannot call `useT()`; they take a `t: Messages`
(or a narrower `labels` sub-tree) parameter threaded in from their calling component.
Their unit tests pass `getMessages('en')` (or the relevant sub-tree) explicitly.

## Boundaries

- The brand name `Tenon` (`src/core/brand.ts`) is passed to messages as the `app`
  param and is **never** translated.
- Not localized: user note content; agent responses (the agent already replies in the
  user's language); the canonical date storage/query language (see
  `date-field-values.md`, `search-query-grammar.md`); format masks (`YYYY/MM/DD`),
  CLI snippets, and env-var names.
- **Pure helpers don't bake in an English fallback.** A helper like `textOf` returns
  the raw node text (`''` for empty); the *display* caller applies the localized
  `|| t.common.untitled`. A baked-in `'Untitled'` silently defeats that localized
  fallback (it is non-empty, so the `||` never fires) — the same trap the agent
  session-title sentinel hit. Helpers that can't reach `t` (pure functions called
  outside React) take the fallback as a parameter, defaulting to the canonical
  English (`DEFAULT_MESSAGES`) so tests and non-localized callers still work.
- Kept English **by design** (not display chrome): strings sent to the agent as
  context (`userViewContext`), clipboard/outline serialization (`selectionActions`),
  generated node *content* written to the document (search-node / capture titles —
  baking the UI language into stored data would be wrong), search-engine internals,
  and the runtime session-title sentinel used for matching.
- The **agent debug panel** (`AgentDebugPanel.tsx`) is a developer diagnostic surface:
  its own chrome (headings, buttons, metric/status/empty-state labels) is localized,
  but the raw protocol it exists to mirror — message role names (`user`/`assistant`),
  part kinds (`tool_call`/`tool_result`/`json`/`thinking`), wire hashes, and the
  token/cost-accounting tooltip — stays verbatim by design.

## Adding a language

Three lock-step edits: extend the `Locale` union + `SUPPORTED_LOCALES`
(`core/locale.ts`), add `core/i18n/messages/<locale>.ts`, and register it in
`LOCALE_OVERRIDES` (`core/i18n/index.ts`). A language may ship partially — untranslated
keys fall back to English, and `tests/core/i18nCoverage.test.ts` validates each file
against the canonical tree and logs per-locale coverage.
